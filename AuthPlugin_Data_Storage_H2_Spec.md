# AuthPlugin 数据存储规范（H2 持久化版本）

## 数据存储方案变更说明

本项目的数据存储方案由原本的 JSON 文件存储，变更为使用 **H2
嵌入式数据库**，并要求数据持久化保存。

------------------------------------------------------------------------

# 一、数据库选型

使用：

-   H2 Database（Embedded 模式）
-   文件持久化模式（非内存模式）
-   JDBC 访问
-   连接池可选（推荐 HikariCP，但必须 shade 并 relocate）

数据库文件必须存储在：

    config/authplugin/database/

例如：

    config/authplugin/database/authplugin.mv.db

------------------------------------------------------------------------

# 二、数据库模式设计

## 1️⃣ Token 表

表名：`tokens`

字段设计：

  字段名         类型      说明
  -------------- --------- -----------------
  token          VARCHAR   主键
  bound_uuid     VARCHAR   绑定的玩家 UUID
  disabled       BOOLEAN   是否禁用
  created_at     BIGINT    创建时间
  last_used_at   BIGINT    最近使用时间

约束： - token 为 PRIMARY KEY - bound_uuid 可为空 - disabled 默认 false

------------------------------------------------------------------------

## 2️⃣ 可扩展字段预留

预留未来扩展字段：

-   ip_address（VARCHAR）
-   expire_at（BIGINT）
-   failed_attempts（INT）

------------------------------------------------------------------------

# 三、持久化要求

1.  必须使用文件模式启动 H2：

        jdbc:h2:file:./config/authplugin/database/authplugin

2.  禁止使用内存数据库模式。

3.  数据必须在服务器重启后完整保留。

4.  必须在插件关闭时安全关闭数据库连接。

5.  不得在主线程执行数据库阻塞操作。

------------------------------------------------------------------------

# 四、线程安全要求

-   所有数据库操作必须通过独立线程池执行。
-   禁止在 Tick 线程直接访问数据库。
-   Token 操作必须事务安全。
-   并发绑定时必须防止重复绑定。

------------------------------------------------------------------------

# 五、DAO 分层要求

必须实现：

-   TokenDao
-   TokenService
-   DatabaseManager

分层规则：

-   DAO 仅负责 SQL
-   Service 负责业务逻辑
-   不允许在 platform 层直接操作数据库

------------------------------------------------------------------------

# 六、初始化流程

插件启动时：

1.  初始化 DatabaseManager
2.  创建数据目录（如不存在）
3.  自动创建数据表（若不存在）
4.  启动 Web 服务

插件关闭时：

1.  安全关闭数据库连接池
2.  释放线程池资源

------------------------------------------------------------------------

# 七、Gradle 要求

-   H2 依赖必须 shade
-   若使用 HikariCP，也必须 shade 并 relocate
-   不得污染 Minecraft 运行时类路径

------------------------------------------------------------------------

# 八、验收标准

1.  数据库文件成功生成
2.  插件重启后 Token 数据仍然存在
3.  并发登录不会产生脏数据
4.  禁用 Token 后立即生效
5.  删除 Token 后无法再自动登录
